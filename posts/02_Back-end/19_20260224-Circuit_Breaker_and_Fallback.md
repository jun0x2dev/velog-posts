# 남의 장애에 휘말리지 않는 법: Circuit Breaker와 Fallback 전략

## 서론
분산 시스템이나 MSA(Microservices Architecture) 환경에서 우리 서비스는 수많은 외부 API와 연결되어 있습니다. 결제, 배송, 알림 등 다양한 외부 서비스를 호출하죠. 그런데 만약 결제 서비스가 갑자기 느려지거나 응답을 멈춘다면 어떻게 될까요? 

우리 서버의 스레드들은 결제 서비스의 응답을 기다리며 하나둘씩 줄을 서기 시작합니다. 결국 모든 스레드가 꽉 차버리고, 결제와 상관없는 로그인이나 상품 조회 기능까지 모두 마비되는 **장애 전파(Cascading Failure)**가 발생합니다.

![도미노처럼 번지는 서비스 장애와 이를 안전하게 차단하는 서킷 브레이커 동작 구조도](/images/02_Back-end/Circuit_Breaker/cascading_failure.png)
*한 서비스의 지연이 전체 시스템의 자원 고갈로 이어지는 위험한 상황입니다.*

이런 대참사를 막기 위해 우리는 전기 회로의 '두꺼비집' 같은 장치가 필요합니다. 오늘은 서비스의 생존을 지켜주는 **서킷 브레이커**와 그 대안인 **폴백**에 대해 알아보겠습니다.

## 본론

### 1. Circuit Breaker: "아픈 곳은 건드리지 말자"
서킷 브레이커는 특정 서비스의 장애가 시스템 전체로 퍼지는 것을 방지하는 **장애 격리 기술**입니다. 장애가 발생한 서비스에 대한 호출을 즉시 차단(**Fail Fast**)하여 시스템의 가용 자원을 보호하고 연쇄 장애를 막는 것이 핵심 목적이죠. 

마치 우리 집 전기 회로가 과부하되면 차단기가 내려가 불이 나는 것을 막아주는 것과 같은 원리입니다.

---

### 2. 핵심 동작 원리: 3가지 상태의 마법
서킷 브레이커는 상황에 따라 세 가지 상태를 오가며 시스템을 지킵니다.

| 상태 | 설명 | 동작 |
| :--- | :--- | :--- |
| **CLOSED (정상)** | 서비스가 건강한 상태입니다. | 모든 요청을 평소처럼 목적지에 전달합니다. |
| **OPEN (차단)** | 에러율이 기준치를 넘은 상태입니다. | 회로를 열어 요청을 즉시 거절하고, 에러나 예비 데이터를 반환합니다. |
| **HALF-OPEN (검사)** | 복구 가능성을 타진하는 상태입니다. | 일정 시간 후 소수의 테스트 요청만 보내봅니다. 성공하면 다시 연결(CLOSED), 실패하면 다시 차단(OPEN)합니다. |

*(기술 다이어그램 설명: CLOSED에서 장애 발생 시 OPEN으로 전이되고, 일정 시간 대기 후 HALF-OPEN 상태에서 복구 여부를 판단하는 상태 전이도)*

---

### 3. 설치 위치와 폴백(Fallback) 전략
서킷 브레이커는 반드시 **호출하는 쪽(Caller)**에 설치해야 합니다. API 게이트웨이나 서비스를 직접 호출하는 상위 서비스가 그 주인공이죠. 

이때 단순히 에러를 띄우지 않고 대안을 제시하는 것을 **우아한 성능 저하(Graceful Degradation)**라고 합니다. 구체적인 폴백 전략은 다음과 같습니다.

- **정적 리소스 및 기본값 반환**: '맞춤형 추천' 엔진이 고장 났다면? 에러 대신 미리 준비된 '실시간 인기 상품 TOP 10'을 보여줍니다. 사용자는 나만의 추천은 못 받지만, 쇼핑은 멈추지 않습니다.
- **캐시된 데이터 활용 (Stale Data)**: 실시간 가격 API가 죽었다면? Redis 등에 캐시된 5분 전 데이터를 보여줍니다. "정보가 최신이 아닐 수 있음"을 안내하더라도 에러 화면보다는 훨씬 낫습니다.
- **해당 영역 숨김 (Feature Hiding)**: 상세 페이지 하단의 '함께 본 상품' 위젯 API가 응답하지 않는다면? 그 영역만 아예 화면에서 지워버립니다. 사용자는 위젯이 있었다는 사실도 모른 채 메인 정보를 읽을 수 있습니다.
- **단순화된 서비스 제공**: 고성능 검색 엔진(Elasticsearch)에 장애가 났다면? 복잡한 필터링은 안 되더라도 DB 직접 조회(LIKE 검색)로 최소한의 검색 기능은 유지합니다.

---

### 4. 실무에서 흔히 저지르는 3가지 실수 (Anti-patterns)
아무리 좋은 도구도 잘못 쓰면 독이 됩니다.

1.  **Retry Storm (재시도 폭풍)**: 설계를 잘못하면 요청이 기하급수적으로 불어납니다.
    - **위험 상황**: API 게이트웨이에서 재시도 3번, 호출되는 서비스 내부에서도 또 재시도 3번을 설정한 경우입니다.
    - **폭격 발생**: 사용자가 버튼을 **딱 1번** 눌렀는데, 시스템 내부에서는 3(게이트웨이) × 3(서비스) = **총 9번의 호출**이 발생합니다.
    - **해결책**: 재시도는 중복으로 하지 말고 반드시 **딱 하나의 계층에서만(Only at one layer)** 관리해야 합니다.
2.  **Timeout Mismatch (타임아웃 불일치)**: 외부 호출 타임아웃을 내부 대기 시간보다 길게 설정하여 시스템을 마비시키는 '좀비 프로세스'를 만드는 경우입니다.
3.  **Too Global Scope (범위 과다)**: 여러 API가 하나의 서킷 브레이커를 공유하여, 특정 서비스 장애 시 상관없는 다른 기능까지 함께 차단되는 경우입니다. 반드시 호출 대상별로 분리해야 합니다.

---

### 5. Resilience4j: 스프링의 수호신
과거에는 Netflix Hystrix를 많이 썼지만, 이제는 **Resilience4j**가 사실상의 표준입니다. 자바 8과 함수형 프로그래밍을 지원하며 매우 가볍습니다.

```java
// Resilience4j 활용 예시
@CircuitBreaker(name = "paymentService", fallbackMethod = "fallbackPayment")
public String processPayment(Order order) {
    return paymentApi.call(order); // 외부 결제 API 호출
}

// 장애 시 실행될 플랜 B
public String fallbackPayment(Order order, Throwable t) {
    log.error("결제 서비스 장애 발생! 사유: {}", t.getMessage());
    return "현재 결제가 지연되고 있습니다. 잠시 후 다시 시도해 주세요.";
}
```

### 6. 실무력을 높이는 결정적 인사이트
서킷 브레이커를 설계할 때 반드시 챙겨야 할 실무 포인트입니다.

1.  **임계치(Threshold) 설정이 중요합니다**: 실패율이 몇 %일 때 문을 닫을지(Failure Rate Threshold), 응답 시간이 몇 초 이상일 때 실패로 간주할지(Slow Call Threshold) 정교하게 설정해야 합니다. 너무 민감하면 멀쩡한 서버를 차단하고, 너무 둔하면 같이 죽습니다.
2.  **모니터링은 필수입니다**: 서킷 브레이커가 열렸다는 것은 시스템에 심각한 문제가 생겼다는 신호입니다. 실시간으로 알림을 받고 원인을 파악할 수 있는 대시보드가 반드시 필요합니다.
3.  **멱등성(Idempotency)을 고려하세요**: 재시도(Retry)를 함께 사용할 경우, 이미 처리된 요청을 다시 보내도 문제가 없는지 확인해야 합니다. 결제 요청을 재시도했는데 중복 결제가 되면 큰일이니까요!

> ### 아키텍트의 시선
> 서킷 브레이커는 단순히 에러를 막는 장치가 아니라, 시스템의 '부분적 가용성'을 확보하는 전략입니다. 모든 기능이 100% 정상일 수는 없지만, 일부가 고장 났을 때 전체가 멈추지 않도록 설계하는 '우아한 성능 저하(Graceful Degradation)'를 항상 염두에 두어야 합니다.

## 결론 및 요약 / 회고
서킷 브레이커는 **"죽은 서비스는 건드리지 않는다"**는 원칙을 실현하는 도구입니다. 분산 시스템에서 장애를 완전히 없앨 수는 없으므로, 장애 발생 시 이를 신속하게 격리하고 시스템이 우아하게 동작하도록 설계하는 것이 무엇보다 중요합니다.

- **장애 격리**: 문제가 생긴 곳을 즉시 도려내어 전체 시스템을 보호하세요.
- **Fail Fast**: 안 될 요청은 빠르게 거절하여 자원 낭비를 막으세요.
- **Fallback**: 최악의 상황에서도 사용자에게 최선의 대안을 제공하세요.

오늘 여러분의 서비스에 튼튼하고 똑똑한 두꺼비집 하나 설치해보는 건 어떨까요?

오늘도 여러분의 서버가 어떤 폭풍우 속에서도 쓰러지지 않길 응원합니다!
