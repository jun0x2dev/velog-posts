# 실시간 통신의 3대장: Polling, SSE, WebSocket 완벽 비교

## 서론
배달 앱에서 라이더의 위치가 실시간으로 움직이는 것을 보거나, 주식 앱에서 초 단위로 변하는 시세를 지켜본 적이 있으신가요? 웹 브라우저는 기본적으로 사용자가 요청을 보내야만 서버가 응답하는 **Request-Response** 구조입니다. 하지만 실시간 서비스에서는 서버가 먼저 "야, 데이터 바뀌었어!"라고 알려줘야 하죠.

![단방향 요청부터 양방향 연결까지 단계별로 발전하는 실시간 통신 기술 진화 구조도](/images/02_Back-end/Real-time_Communication/communication_evolution.png)
*단방향 요청부터 양방향 실시간 연결까지, 웹 통신 기술의 발전 단계입니다.*

전통적인 HTTP의 한계를 극복하고 실시간성을 확보하기 위해 우리는 여러 가지 꼼수를 써왔습니다. 오늘은 가장 원초적인 방법인 **Polling**부터, 현대적인 **WebSocket**까지 각각의 특징과 어떤 상황에 써야 할지 알아보겠습니다.

## 본론

### 1. Polling: "나왔어? 아직? 지금은?"
가장 고전적이고 단순한 방법입니다. 클라이언트가 주기적으로(예: 5초마다) 서버에 "새로운 데이터 있어?"라고 물어보는 방식입니다.

- **특징**: 단순한 HTTP 요청입니다. 특별한 설정이 필요 없죠.
- **문제점**: 데이터가 변하지 않았는데도 계속 요청을 보내니 서버 자원이 낭비됩니다. 응답을 받은 직후에 데이터가 변하면 다음 요청 때까지 기다려야 하므로 실시간성이 떨어집니다.

![클라이언트가 주기적으로 서버에 데이터를 요청하고 응답받는 단순한 폴링 동작 구조도](/images/02_Back-end/Real-time_Communication/polling_mechanism.png)

---

### 2. SSE (Server-Sent Events): "서버가 보내주는 라디오 방송"
서버에서 클라이언트로 데이터를 일방적으로 밀어주는(Push) 방식입니다. 

- **특징**: 한 번 연결을 맺으면 서버가 필요할 때마다 데이터를 보냅니다. HTTP 프로토콜을 그대로 사용하며 가볍습니다.
- **인사이트**: 주식 시세나 알림 서비스처럼 **서버에서 클라이언트로만** 데이터가 흐르는 경우에 최적입니다. 재연결 기능이 내장되어 있어 안정적입니다.

![서버가 연결된 클라이언트에게 일방적으로 데이터를 계속해서 밀어주는 효율적인 SSE 동작 구조도](/images/02_Back-end/Real-time_Communication/sse_mechanism.png)

---

### 3. WebSocket: "전화기처럼 주고받는 양방향 통로"
HTTP의 한계를 완전히 벗어나, 클라이언트와 서버가 서로 자유롭게 데이터를 주고받는 전용 통로를 만드는 방식입니다.

- **특징**: 한 번 연결(Handshake)되면 연결을 끊기 전까지 계속 유지됩니다. HTTP보다 헤더가 훨씬 가벼워 대량의 데이터를 빠르게 주고받을 수 있습니다.
- **인사이트**: 채팅, 실시간 게임, 협업 툴(Google Docs 등)처럼 **양방향 소통**이 빈번한 서비스에 필수적입니다.

![클라이언트와 서버가 전용 통로를 통해 자유롭게 양방향 데이터를 주고받는 강력한 웹소켓 동작 구조도](/images/02_Back-end/Real-time_Communication/websocket_mechanism.png)

### 4. 한눈에 비교하는 실시간 통신 전략

| 방식 | 통신 방향 | 실시간성 | 서버 부하 | 주요 용도 |
| :--- | :--- | :--- | :--- | :--- |
| **Polling** | 단방향 (C->S) | 낮음 | 매우 높음 | 단순 알림, 트래픽 적은 기능 |
| **SSE** | 단방향 (S->C) | 높음 | 중간 | 뉴스 피드, 주식 시세, 알림 |
| **WebSocket** | **양방향** | **매우 높음** | 낮음 (연결 유지 비용 발생) | 채팅, 게임, 실시간 협업 |

---

### 5. 실무력을 높이는 결정적 인사이트
실무에서 기술을 선택할 때 반드시 고려해야 할 포인트입니다.

1.  **무조건 WebSocket이 정답은 아닙니다**: WebSocket은 서버와 연결을 계속 유지해야 하므로, 동시 접속자가 많아지면 서버 메모리 부하가 상당합니다. 단순히 서버 알림만 필요하다면 훨씬 가벼운 **SSE**가 더 나은 선택일 수 있습니다.
2.  **로드 밸런싱을 주의하세요**: WebSocket은 상태를 유지(Stateful)합니다. 서버가 여러 대라면 사용자가 처음에 접속했던 서버와 계속 통신해야 하므로 **Sticky Session** 설정이 필요합니다.
3.  **대안으로 웹훅(Webhook)도 고려하세요**: 서버와 서버 간의 실시간 데이터 전달이라면 폴링보다는 상대방의 엔드포인트를 직접 찌르는 웹훅이 표준입니다.

---
### 6. 실전 케이스 스터디: "대규모 티켓팅 대기열 시스템"
실시간 통신 기술이 가장 빛을 발하는 곳은 바로 대규모 트래픽이 몰리는 대기열 시스템입니다. 단순히 순서를 지키는 것을 넘어 사용자에게 **"현재 대기 순서: 5,320번째"**라는 피드백을 실시간으로 주는 것이 핵심이죠. 여기서 우리는 인프라 선택의 기로에 서게 됩니다.

#### MQ(Kafka)의 한계 vs Redis ZSET의 승리
보통 대용량 처리는 MQ(Message Queue)가 정답이라고 생각하기 쉽지만, '실시간 순번 조회'가 필요한 대기열에서는 이야기가 달라집니다.

- **MQ의 한계**: 큐는 파이프 구조입니다. 메시지가 소비되어야만 사라지며, 중간에 있는 특정 사용자가 몇 번째인지 조회(Random Access)하는 것은 매우 어렵습니다.
- **Redis ZSET의 강점**: 대기열 자체를 데이터로 관리합니다. `ZRANK` 명령어를 통해 특정 유저의 등수를 $O(\log N)$ 속도로 즉시 확인할 수 있습니다.

![사용자별 실시간 대기 순번을 효율적으로 관리하는 Redis Sorted Set 기반 대기열 구조도](/images/02_Back-end/Real-time_Communication/redis_zset_queue.png)

| 구분 | MQ (Kafka/RabbitMQ) | Redis (Sorted Set) |
| :--- | :--- | :--- |
| **주 목적** | 비동기 처리, 버퍼링 | 실시간 랭킹, 상태 관리 |
| **순번 조회** | 매우 어려움 | **즉시 가능 ($O(\log N)$)** |
| **중복 방지** | 별도 로직 필요 | **Key 기반 자동 해결 (멱등성)** |
| **임의 취소** | 중간 삭제 어려움 | **즉시 가능 (ZREM)** |

#### 하이브리드 아키텍처: Redis와 MQ의 조화
실무에서는 이 두 기술을 대립시키는 것이 아니라 역할에 맞게 혼합하여 사용합니다.

1. **앞단 (Waiting Phase) - Redis**: 조회와 관리가 중요하므로 ZSET으로 사용자 대기 순번을 관리하고 실시간 등수를 보여줍니다.
2. **뒷단 (Processing Phase) - MQ**: 실제 처리가 시작되는 단계(예: 티켓 예매 처리, DB 저장)에서는 높은 처리량과 비동기 처리를 위해 Kafka나 RabbitMQ로 데이터를 넘겨 처리합니다.

## 결론 및 요약 / 회고
실시간 통신 기술에 있어 '절대적인 정답'은 없습니다. 우리가 마주한 시스템의 규모, 예상 트래픽, 그리고 허용 가능한 지연 시간(Latency)에 따라 최적의 선택지는 매번 달라지기 때문입니다.

- **Polling**: 구현이 매우 쉽지만 자원 낭비가 심하므로, 트래픽이 적은 내부 관리용 기능에 적합합니다.
- **SSE**: 서버에서 클라이언트로의 일방향 알림이 주된 목적일 때 가장 가볍고 효율적인 대안입니다.
- **WebSocket**: 긴밀한 양방향 소통이 필수적인 채팅이나 게임에 강력하지만, 연결 유지 비용과 서버 확장성을 심도 있게 고려해야 합니다.

중요한 것은 "우리 서비스의 비즈니스 도메인이 진정으로 필요로 하는 것이 무엇인가?"를 정확히 꿰뚫어 보는 통찰력입니다.

> ### 아키텍트의 시선
> 실시간 서비스 설계의 진정한 묘미는 통신 프로토콜을 넘어선 **'인프라와 전략의 조합'**에 있습니다. 예를 들어 수백만 명이 동시에 접속하는 **대규모 티켓팅 서비스**에서 실시간 대기 순번을 알려줘야 한다면, 단순히 웹소켓을 쓰는 것을 넘어 서버 간 메시지 전파를 위한 브로커 선택이 핵심이 됩니다. 이때 데이터의 영속성보다 **극도의 저지연(Low Latency)**과 처리 속도가 중요하다면 Kafka보다는 메모리 기반의 **Redis Pub/Sub**이 훨씬 날카로운 선택지가 될 수 있습니다. 반면, 메시지의 유실이 절대 없어야 하고 대량의 로그성 이벤트를 순차적으로 처리해야 한다면 Kafka가 정답일 것입니다. 기술의 이름에 매몰되지 말고, 그 기술이 해결하고자 하는 **'문제의 본질'**을 먼저 바라보시기 바랍니다.

오늘도 여러분의 패킷이 지연 없이 사용자에게 도달하길 응원합니다!
