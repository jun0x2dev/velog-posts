# 읽기와 쓰기의 분리: CQRS 패턴으로 확장성 있는 시스템 설계하기

## 서론
우리가 처음 웹 개발을 배울 때 만드는 `MemberService`는 보통 하나의 모델로 가입도 하고, 수정도 하고, 조회도 합니다. 하지만 서비스가 커지면 문제가 생깁니다. 가입할 때는 이름과 이메일만 있으면 되는데, 통계 화면이나 마이페이지에서는 회원의 구매 내역, 포인트, 활동 로그까지 온갖 데이터를 한꺼번에 조회해야 하죠.

하나의 엔티티에 수정 로직과 조회 로직이 뒤섞이면 코드는 비대해지고, 조회 성능을 높이려고 인덱스를 걸면 수정 성능이 떨어지는 딜레마에 빠집니다. 오늘은 "책임을 분리하면 시스템이 건강해진다"는 철학을 담은 **CQRS** 패턴을 파헤쳐 보겠습니다.

![명령과 조회의 책임이 계층별로 명확히 나누어진 효율적인 CQRS 아키텍처 구조도](/images/02_Back-end/CQRS_Pattern/cqrs_1.png)
*하나의 모델에 섞여 있던 명령과 조회 로직을 서비스와 리포지토리 계층에서 물리적으로 분리하는 구조적 변화입니다.*

## 본론

### 1. CQRS란 무엇인가? (용어 정리)
CQRS는 **Command Query Responsibility Segregation**의 약자로, 우리말로 번역하면 **'명령과 조회의 책임 분리'**입니다.

- **Command (명령)**: 시스템의 상태를 변경하는 작업 (Create, Update, Delete). "데이터를 어떻게 바꿀 것인가?"에 집중합니다.
- **Query (조회)**: 시스템의 상태를 반환하는 작업 (Read). "데이터를 어떻게 보여줄 것인가?"에 집중합니다.

단순히 클래스를 나누는 것을 넘어, 데이터를 바라보는 관점 자체를 두 개로 쪼개는 것이 핵심입니다.

---

### 2. 단계별 CQRS 적용 전략

#### ① 코드 서비스 로직에서의 분리 (Beginner)
DB는 하나를 쓰되, 소스 코드 레벨에서만 명령 서비스(`MemberCommandService`)와 조회 서비스(`MemberQueryService`)를 분리하는 방식입니다.

- **방법**: 수정 로직이 담긴 Service와 단순 조회(DTO 반환)용 Service를 나눕니다.
- **장점**: 코드의 **단일 책임 원칙(SRP)**을 지킬 수 있습니다. 수정 로직에 복잡한 JPA 연관관계가 얽혀 있어도, 조회 로직은 전용 DTO와 QueryDSL을 사용하여 성능을 최적화할 수 있습니다. 트랜잭션 충돌 범위가 줄어드는 효과도 있습니다.

#### ② 데이터베이스 수준의 분리 (Advanced)
쓰기 전용 DB와 조회 전용 DB를 물리적으로 나누는 방식입니다.

- **Read Replica (Master-Slave)**: 쓰기는 Master DB에서, 읽기는 여러 대의 Slave DB에서 처리합니다. 가장 일반적인 성능 확장 방법입니다.
- **Polyglot Persistence**: 쓰기는 관계형 DB(RDB)에, 조회는 검색에 특화된 NoSQL(Elasticsearch)이나 캐시(Redis)에 저장합니다.

---

### 3. 실무에서의 DB 분리와 동기화 전략
실무에서 고성능 시스템을 구축할 때 DB 분리는 필수적입니다. 이때 가장 중요한 것이 **'두 DB 사이의 데이터를 어떻게 맞출 것인가'**입니다.

![DB 분리형 CQRS와 이벤트 연동](/images/02_Back-end/CQRS_Pattern/cqrs_database_separation.png)
*쓰기 DB의 변화를 이벤트를 통해 조회 전용 DB로 전파하는 구조입니다.*

1. **이벤트 발행**: `MemberCommandService`에서 정보를 수정하면 `MemberUpdatedEvent`를 발행합니다. (지난 포스팅에서 다룬 `@TransactionalEventListener`가 여기서 쓰입니다!)
2. **메시지 브로커**: Kafka나 RabbitMQ를 통해 이벤트를 전달합니다.
3. **데이터 가공 및 저장**: 조회 전용 서비스가 이벤트를 구독하여, 조회에 최적화된 형태로 데이터를 가공한 뒤 조회용 DB(예: Elasticsearch)에 밀어 넣습니다.

---

### 4. CQRS의 명암: 장점과 단점

| 장점 (Pros) | 단점 (Cons) |
| :--- | :--- |
| **독립적 확장**: 조회가 많은 서비스에서 조회 서버/DB만 무한히 늘릴 수 있습니다. | **복잡도 증가**: 유지보수해야 할 코드와 인프라(메시지 브로커 등)가 늘어납니다. |
| **성능 최적화**: 조회 시 Join을 없애고 미리 가공된 데이터를 읽기만 하면 됩니다. | **결과적 일관성**: 쓰기 DB와 조회 DB의 데이터가 일치하기까지 아주 미세한 시차가 발생합니다. |
| **도메인 집중**: 쓰기 모델은 비즈니스 규칙에만 집중할 수 있어 코드가 깔끔해집니다. | **설계 비용**: 초기 설계와 구현에 더 많은 시간과 노력이 들어갑니다. |

---

### 5. 실무 적용 가이드: 언제 사용해야 할까?
CQRS는 비싼 도구입니다. 다음과 같은 상황에서만 도입을 고려하세요.

1. **읽기 요청이 쓰기보다 압도적으로 많을 때** (예: 이커머스 상품 조회 vs 주문)
2. **조회 시 요구되는 데이터 가공이 너무 복잡할 때** (예: 여러 테이블을 조인해야 하는 대시보드나 통계)
3. **도메인 로직이 너무 복잡해서 조회 로직과 섞이면 관리가 안 될 때**

## 결론 및 요약 / 회고
CQRS는 단순히 기술적인 패턴을 넘어, **"사용자의 의도(명령)와 결과(조회)를 분리하여 생각하는 철학"**입니다.
- 작은 서비스라면 **코드 레벨**에서의 분리만으로도 충분한 이점을 얻을 수 있습니다.
- 대규모 서비스라면 **DB 분리**와 **이벤트 기반 동기화**를 통해 한계를 돌파하세요.

처음부터 모든 곳에 CQRS를 적용하려 하지 마세요. 가장 고통받고 있는 '조회 성능 지옥'부터 하나씩 분리해 나가는 것이 프로 개발자의 지혜입니다.

> ### 아키텍트의 시선
> CQRS를 도입할 때 가장 큰 적은 '기술적 욕심'입니다. 무작정 DB를 분리하기 전에, 먼저 서비스 레이어에서 명령과 조회를 나누는 것부터 시작해 보세요. 그것만으로도 여러분의 코드는 훨씬 숨통이 트일 것입니다. **"충분히 복잡해졌을 때 도입해도 늦지 않습니다."**

오늘도 여러분의 시스템이 복잡함을 딛고 명쾌하게 돌아가길 응원합니다!
