# RDB vs NoSQL 기술 선택의 기준: 서비스 성격에 따른 데이터베이스 설계 전략

## 서론
새로운 프로젝트를 시작할 때 백엔드 개발자가 가장 먼저 직면하는 고뇌의 순간이 있습니다. "데이터베이스는 뭘로 하지?" 

누군가는 "전통의 강자 MySQL이 최고지!"라고 하고, 누군가는 "요즘은 유연한 MongoDB가 대세야!"라고 외칩니다. 마치 짜장면과 짬뽕 사이에서 고민하는 것처럼, RDB와 NoSQL 사이에서의 선택은 서비스의 운명을 결정짓는 중요한 결정입니다. 

![RDB vs NoSQL 선택 가이드라인](/images/04_Database/RDB_vs_NoSQL_Selection_Guide/db_selection_matrix.png)
*데이터의 정합성, 확장성, 스키마 유연성 등 서비스 요구사항에 따른 최적의 DB 선택 기준을 시각화한 매트릭스입니다.*

오늘은 이 고민을 한 방에 해결해 줄 기준을 정리해 보겠습니다. 단순히 기술의 장단점을 나열하는 것을 넘어, **언제 어떤 상황에서** 무엇을 꺼내 들어야 우리 본체가 평온할 수 있을지 파헤쳐 보시죠.

## 본론

### 1. RDB (관계형 데이터베이스): "질서와 정확성이 생명입니다"
MySQL, PostgreSQL, Oracle 등이 속한 RDB는 데이터를 테이블, 행, 열의 구조로 엄격하게 관리합니다.

- **강점**: 데이터의 정확성(ACID)이 최우선입니다. 복잡한 JOIN이 가능하며, 데이터 구조(Schema)가 명확합니다.
- **언제 쓸까?**: 금융 시스템, 전자상거래(결제), 회원 정보 관리 등 **데이터의 무결성**이 절대적으로 중요한 경우.

---

### 2. NoSQL (비관계형 데이터베이스): "유연함과 확장이 무기입니다"
MongoDB, Redis, Cassandra 등이 속한 NoSQL은 고정된 스키마 없이 데이터를 문서, 키-값, 그래프 등 다양한 형태로 저장합니다.

- **강점**: 수평적 확장(Scale-out)이 매우 쉽습니다. 데이터 구조가 수시로 바뀌어도 유연하게 대응할 수 있고, 읽기/쓰기 속도가 매우 빠릅니다.
- **언제 쓸까?**: 로그 데이터, 소셜 미디어의 피드, 실시간 랭킹, 대규모 트래픽이 발생하는 비정형 데이터 처리 등.

---

### 3. 기술 선택의 핵심 지표: CAP 이론
NoSQL을 이해하는 가장 중요한 이론입니다. 분산 시스템에서 다음 세 가지를 모두 만족하는 것은 불가능하다는 정리죠.

- **Consistency (일관성)**: 모든 노드가 동시에 같은 데이터를 보여줌.
- **Availability (가용성)**: 일부 노드가 죽어도 시스템은 계속 응답함.
- **Partition Tolerance (분할 내성)**: 네트워크가 끊겨도 시스템이 동작함.

**인사이트**: 대부분의 NoSQL은 **CP** 또는 **AP** 중 하나를 선택합니다. 반면 RDB는 전통적으로 **CA**를 지향하죠. 우리 서비스가 "데이터가 조금 늦게 반영되어도 24시간 떠 있어야 하는가(AP)?" 아니면 "무조건 정확한 데이터를 보여줘야 하는가(CP)?"를 고민해야 합니다.

| 특징 | RDB (SQL) | NoSQL |
| :--- | :--- | :--- |
| **스키마** | 엄격함 (Fixed Schema) | 유연함 (Dynamic Schema) |
| **확장 방식** | 수직적 (Scale-up) | 수평적 (Scale-out) |
| **데이터 관계** | JOIN으로 연결 | 데이터 내부에 포함 (Embedded) |
| **트랜잭션** | 강력한 ACID 보장 | 대부분 BASE (최종 일관성) |

---

### 4. 실무력을 높이는 결정적 인사이트
현대 아키텍처에서는 둘 중 하나만 선택하지 않습니다. **Polyglot Persistence**가 정답입니다.

1.  **섞어 쓰세요**: 메인 데이터(결제, 회원)는 PostgreSQL(RDB)에 저장하고, 실시간 알림이나 캐시 데이터는 Redis(NoSQL)에, 로그 데이터는 MongoDB(NoSQL)에 저장하는 식입니다.
2.  **데이터 모델링의 차이**: RDB는 "어떻게 저장할까(Normalization)"를 고민하고, NoSQL은 "어떻게 읽어올까(Query-driven)"를 먼저 고민해야 합니다. 읽기 패턴에 맞춰 데이터를 중복 저장하는 것이 NoSQL의 미학입니다.
3.  **성급한 최적화는 금물**: 처음부터 "트래픽이 많을 테니 NoSQL로 가자!"라고 하기보다, RDB로 시작해서 한계가 느껴질 때 특정 부분만 NoSQL로 분리하는 것이 가장 안전한 전략입니다.

## 결론 및 요약 / 회고
RDB와 NoSQL은 서로 적이 아니라, 서로의 부족한 점을 채워주는 파트너입니다.
- **데이터의 정확성**과 **복잡한 관계**가 중요하다면 **RDB**.
- **방대한 데이터**와 **빠른 확장성**이 중요하다면 **NoSQL**.
- 결국은 서비스의 특성에 맞춰 **적재적소에 섞어 쓰는 능력**이 백엔드 개발자의 핵심 역량입니다.

여러분의 프로젝트가 어떤 데이터베이스 위에서 가장 아름답게 꽃필 수 있을지, 오늘 배운 기준들로 다시 한번 점검해 보시길 바랍니다.
